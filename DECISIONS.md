## ADR-001 — Canvas 2D API over Satori for client-side rendering

**Status:** Accepted
**Date:** 2026-07-07

### Context

Satori (Vercel) can render React JSX to SVG but requires a Node.js/Edge runtime — incompatible with pure static export. An alternative is to render HTML in an iframe and screenshot it, but this is fragile. Canvas 2D API works natively in every modern browser.

### Decision

Use Canvas 2D API (drawImage, fillText, fillRect, createLinearGradient) in ogRenderer.ts. All rendering is synchronous and deterministic.

### Consequences

No React rendering of templates — raw canvas draw calls. Manual text wrapping required (wrapText helper). More control over exact pixel output. Works completely offline.

## ADR-002 — Pre-populated default content — canvas never blank

**Status:** Accepted
**Date:** 2026-07-07

### Context

A blank canvas on first load looks broken and fails to demonstrate the product value. Users need to see an example immediately.

### Decision

Pre-populate title="Your Project Name" and description="The best tool for developers." so the canvas always has content on first render.

### Consequences

Users see a working example immediately. Better first impression. Copy is clearly placeholder — users know to replace it.

## ADR-003 — Single-page tool — no routing

**Status:** Accepted
**Date:** 2026-07-07

### Context

All functionality fits on one page. Multi-page would add navigation complexity with no user benefit.

### Decision

Single route (/) only. Static export. Follow ImageCompress architecture exactly.

### Consequences

Simplest possible architecture. Fast cold load. Easy to extend to multi-page if needed for v0.2 template gallery.

## ADR-004 — Reuse download.ts pattern from ImageCompress

**Status:** Accepted
**Date:** 2026-07-07

### Context

ImageCompress already has a proven download pattern: generateFilename() + downloadBlob(). PNG download from canvas uses the same Blob + anchor click approach.

### Decision

Copy and adapt download.ts from ImageCompress. Change prefix from "imagecompress-" to "og-".

### Consequences

Zero new code for download logic. Consistent filename pattern across all DevOS tools.
